Player & State Machine

The player character's movement and actions are governed by a robust state machine. Rather than packing all conditions into a single monolithic script, behaviors are distributed across discrete, isolated states.

⚠️ Class Name Notice

The source file is named PlayerController.cs, but the main MonoBehavior class inside it is named Player.

State Machine Architecture

The player uses the classes PlayerStateMachine, PlayerState, and Player to manage transitions. The three main states are:

PlayerIdleState

File: PlayerIdleState.cs

The default resting state. The player is stationary. The system monitors for mouse clicks to initiate slow-motion aiming.

  • Transition: Left-click → PlayerAimingState.

PlayerAimingState

File: PlayerAimingState.cs

Activated when pulling back the slingshot. Controls time-dilation and trajectory simulation.

  • Enters slow motion by scaling time: Time.timeScale = 0.1f * player.AimingTimeScaleModifier.
  • Saves drag starting location.
  • Computes drag direction vector: (startPos - currentMousePos).normalized.
  • Simulates trajectory paths dynamically in Update().
  • Transition on Release:
    • If drag distance > 0.5f → PlayerDashingState.
    • If drag distance ≤ 0.5f → PlayerIdleState (Cancel aim).

PlayerDashingState

File: PlayerDashingState.cs

The high-speed movement state. Handles collisions, speed decay, and active mutations.

  • Resets bounce counter and launches the player.
  • If the Double Dash mutation is active, pressing left-click mid-dash will consume 5s of life time and return the player to PlayerAimingState for redirection.
  • Monitors speed in Update(). Returns to PlayerIdleState when linear velocity drops below 0.15f after a minimum duration (0.15s).
  • Applies active mutations in FixedUpdate():
    • Vacuum Blade: Pulls nearby enemies towards player path.
    • Trail of Fire: Spawns damaging fire triggers.

Core Mechanics & Math

Slingshot Launch Physics

Launched force is calculated inside the LaunchPlayer() method in Player.cs. It calculates the displacement between drag start and end positions, clamps the magnitude to prevent excessive speeds, and applies an impulse force:

// 1. Calculate force vector (slingshot action: start - end)
Vector2 forceVector = startPosition - endPosition;

// 2. Clamp to weapon's maximum allowed drag distance
Vector2 clampedForce = Vector2.ClampMagnitude(forceVector, currentWeapon.maxDragDistance);

// 3. Scale with weapon power multipliers
float finalPower = basePower * currentWeapon.powerMultiplier;

// 4. Apply instantaneous impulse
rb.AddForce(clampedForce * finalPower, ForceMode2D.Impulse);

Trajectory Prediction & Reflection Math

The DrawTrajectory(Vector2 startPos, Vector2 direction) method calculates bounce lines recursively (up to 2 bounces) utilizing 2D raycasts and mathematical vector reflections:

// 1. First Raycast check (Start -> Wall 1)
RaycastHit2D hit1 = Physics2D.Raycast(startPos, direction, 50f, collisionLayers);
if (hit1.collider != null) {
    lr.SetPosition(1, hit1.point);
    
    // 2. Compute reflection vector
    Vector2 reflectDirection = Vector2.Reflect(direction, hit1.normal);
    
    // 3. Second Raycast check (Wall 1 -> Wall 2)
    RaycastHit2D hit2 = Physics2D.Raycast(hit1.point + reflectDirection * 0.1f, reflectDirection, 50f, collisionLayers);
    if (hit2.collider != null) {
        lr.SetPosition(2, hit2.point);
    } else {
        lr.SetPosition(2, hit1.point + reflectDirection * 10f);
    }
}

Bounces and Collision Checks

Wall bounces are registered inside OnCollisionEnter2D of the Player component. It checks if the collided object name contains "Wall". The bounce count (CurrentDashBounces) is stored, which is utilized by upgrades like Kinetic Momentum to scale damage output.
